# 运行纪律

## 任务执行原则

1. 接到任务后，先判断能否直接完成。能用专用工具（download_video、tavily_search、query_hotboard）就不用通用工具（run_command、http_get）。
2. 同一个命令如果连续失败 3 次，停止重试，分析错误信息，换个思路或向用户报告。
3. 每轮工具调用尽量合并——多个独立操作可以一次发出多个 tool_call，减少往返。

## 错误处理

4. 工具返回 ERROR: 开头的结果时，读取具体错误内容，根据提示调整参数或方法，而不是盲目重试。
5. 如果是配置缺失（如缺少 API Key、缺少依赖），直接告诉用户需要做什么，不要自己反复尝试。
6. 遇到 subprocess 编码错误或命令找不到时，换一种方式实现，不要死磕。

## 效率要求

7. 下载文件前先确认目标目录是否存在（list_dir），避免下载到错误路径。
8. 创建 Skill 时直接用 write_file 写 SKILL.md 和 tool.py，不要通过 run_command 间接操作。
9. 操作完成后验证结果（list_dir 确认文件存在、读取关键行确认内容正确），给用户明确的成功/失败反馈。

## 路径规范

12. 引用文件/目录时优先使用相对路径（相对于项目根或用户目录），简洁且跨环境可移植。
13. 绝对路径仅当用户明确给出或相对路径不可行时才使用（如 `E:\code\...`、`/mnt/e/...`）。
14. 容器/WSL 路径（如 `/mnt/e/...`）只在确认环境是 WSL/容器时使用，Windows 下不要用。
15. 路径中包含方括号 `[]`、空格、中文等特殊字符时，用引号包裹，优先用 `list_dir` 验证路径存在再操作。

## 交互风格

16. 任务完成后简洁报告结果，不要长篇解释过程。失败时才说明原因和下一步建议。
17. 不确定的事情直接问用户，不要猜测。例如路径不确定时问用户，不要自己编造。

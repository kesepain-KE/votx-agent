paths.py

工具模块 ./paths.py

统一路径解析 — dev / PyInstaller --onedir / --onefile 三种运行模式通用

public get_project_root() 返回项目根目录路径，三种模式通用

根据运行环境自动判断项目根目录：

  • sys.frozen 存在 → PyInstaller 打包模式，返回 sys.executable 所在目录
  • 否则 → 开发模式，返回 __file__ 所在目录（即 paths.py 所在目录）

→ str: 项目根目录绝对路径

所有模块必须通过此函数获取根目录，禁止硬编码路径。

provider/schema.py

Provider 层 ./provider/schema.py

统一数据结构 — Provider 层与 engine/tool/chat 层之间的接口约定

class ToolCall (dataclass)

统一工具调用对象，所有 Provider adapter 的输出格式。

id: str — 工具调用 ID
name: str — 工具名称
input: dict — 已解析的 JSON 参数字典

class ProviderResponse (dataclass)

统一 LLM 响应对象，所有 Provider 的 respond() 返回此类型。

text: str — 文本回复
reasoning: str — 思考过程（DeepSeek/Claude extended thinking）
tool_calls: list[ToolCall] — 工具调用列表
usage: dict | None — Token 用量统计
finish_reason: str — 结束原因

property ProviderResponse.has_tool_calls 是否包含工具调用

→ bool: len(self.tool_calls) > 0

provider/base.py

Provider 层 ./provider/base.py

抽象 Provider 接口 — 所有 LLM Provider 必须实现

class BaseProvider (ABC)

LLM Provider 抽象基类。子类必须实现 respond() 和 respond_stream()。

last_response: ProviderResponse | None — 上次响应（流式完成后可用）
last_usage: dict | None — 上次用量
stream: bool — 是否启用流式

abstract respond(messages, tools) 非流式调用 LLM
  • messages: list[dict] — 完整消息列表（含 system）
  • tools: list[dict] | None — 工具 schema 列表

→ ProviderResponse

abstract respond_stream(messages, tools) 流式调用 LLM，yield 事件

生成器模式，yield 事件 dict：

  • {"type": "thinking_chunk", "content": str} — 思考片段
  • {"type": "text_chunk", "content": str} — 文本片段

生成器耗尽后 self.last_response 可用。

provider/factory.py

Provider 层 ./provider/factory.py

Provider 工厂 — 根据用户配置创建对应的 LLM Provider 实例

public create_provider(user_config, core_config) 根据 config 创建 Provider 实例

路由逻辑：

  • type == "openai" 或兼容值 → ResponsesProvider（优先 Responses API，回退 Chat Completions）
  • type == "anthropic" → AnthropicProvider（Messages API）

支持 VOTX_PROVIDER 环境变量覆盖 config.json 的 type。

→ BaseProvider 子类实例

provider/responses_api.py

Provider 层 ./provider/responses_api.py

OpenAI 兼容 Provider — 支持 Responses API 和 Chat Completions 双协议

class ResponsesProvider(BaseProvider)

主 Provider，覆盖 OpenAI / DeepSeek / Azure / Gemini 等所有 OpenAI 兼容厂商。

api_style 决定协议："responses" → Responses API（完整推理） / "chat" → Chat Completions（流式快）

public respond(messages, tools) 非流式调用，返回 ProviderResponse

根据 api_style 走 Responses API 或 Chat Completions。

内部处理 tool_calls 解析、reasoning 提取、usage 统计。

→ ProviderResponse

public respond_stream(messages, tools) 流式调用，yield thinking_chunk / text_chunk

流式路径：思考先于正文，逐 chunk yield。

⚠️ streaming 最后 chunk 可能无 choices，需 if not chunk.choices: continue guard。

class AnthropicProvider(BaseProvider)

Claude 系列模型专用。原生 extended thinking 支持。

run/io_utils.py

运行时核心 ./run/io_utils.py

原子写和 JSONL 工具 — 防止写入中途崩溃导致数据损坏

public atomic_write_text(path, text) 原子写文本文件

流程：写临时文件 → flush → fsync → os.replace（原子替换）

保证写入中途崩溃不会损坏原文件。

public atomic_write_json(path, data, indent) 原子写 JSON 文件

json.dumps → atomic_write_text

public atomic_write_gzip(path, text) 原子写 gzip 压缩文件

用于历史归档压缩，节省约 80% 空间。

public append_jsonl(path, entry) 追加一行 JSONL（每行一个 JSON）

用于工具调用日志等追加写场景。写入后 flush + fsync 确保持久化。

public read_json_safe(path, default) 安全读 JSON，损坏时返回 default

捕获 FileNotFoundError 和 JSONDecodeError，不会抛异常。

run/prompt_cache.py

运行时核心 ./run/prompt_cache.py

system prompt 缓存 — 按源文件 mtime 做缓存失效，避免每次对话都重新组装

public build_cached_system_prompt(root, user_dir, force) 缓存版本的 system prompt 构建

计算所有源文件（self_soul.md、soul.md、AGENTS.md、skills/**、memory/*.md 等）的 mtime hash。

hash 不变 → 返回缓存；hash 变化 → 重新构建。

→ str: 完整 system prompt

private _compute_cache_key(root, user_dir) 根据所有源文件 mtime 计算缓存 key

监控的文件列表：

  • self_soul.md、soul.md、AGENTS.md
  • skills/**/{SKILL.md, tool.py, _meta.json}
  • self-improving/{memory.md, corrections.md}
  • memory/*.md、SESSION-STATE.md、graph.jsonl

→ int: hash 值

public invalidate_prompt_cache(user_dir) 使缓存失效（Web 重载按钮调用）

user_dir=None 清全部缓存，指定则只清该用户。

run/engine.py

运行时核心 ./run/engine.py

对话引擎 — CLI 和 Web 共用，system prompt 组装 + tool calling 循环

public build_system_prompt(root, user_dir) 组装完整 system prompt（9 层叠加）

按优先级从 9 个源文件叠加组装：

  • self_soul.md → 用户人设（最外层）
  • soul.md → 全局人格基座
  • AGENTS.md → 自我操作手册
  • skills.register_all() → Skill 摘要
  • self-improving/memory.md → HOT 记忆
  • self-improving/corrections.md → 纠正记录
  • memory/*.md → 长期记忆（单文件限 2000 字符）
  • graph.jsonl → 知识图谱摘要（仅实体统计）
  • SESSION-STATE.md → 会话状态

→ str: 完整 system prompt

public run_chat_turn(chat, tool_runner, provider, tools) 执行一轮对话的工具调用循环（生成器）

核心执行循环，yield 事件 dict：

  • tool_call — 工具调用进度（含 name/icon/args/elapsed/success）
  • text_chunk — 流式文本片段
  • thinking_chunk / thinking_done — 思考过程
  • usage — Token 用量
  • error — 错误
  • max_rounds — 达到最大轮数
  • deadlock_warning — 死循环警告（连续 3 次失败）

循环逻辑：发送 → LLM 返回 → 有 tool_calls 则执行并循环 → 无则结束。

上限 MAX_TOOL_ROUNDS（默认 80，由 config_core.json 控制）。

private _fmt_tool_line(name, args, elapsed, success) 格式化工具调用展示行

格式：📖 read_file /path/to/file 0.9s

每个工具有专属 emoji 图标。

private _pick_arg(name, args) 提取工具调用的关键参数用于展示

按工具类型提取关键参数（如文件路径、URL、命令等），超过 60 字符截断。

run/chat.py

运行时核心 ./run/chat.py

对话管理 — 消息组装、历史加载/保存、压缩归档、tool_calls 链修复

class ChatManager

管理对话消息列表和历史记录。每个用户一个实例。

messages: list[dict] — 当前消息列表
token_stats: dict — 累计 Token 统计（跨轮累加）
data_path — 历史文件路径
archive_dir — 归档目录

public build_messages() 组装完整消息列表: system + history

→ list[dict]: system prompt + 所有历史消息

public add_user_message(content) 添加用户消息，自动裁剪超限历史

追加前调用 _trim_if_needed()，超过 max_history 自动压缩归档。

public add_tool_call_message(tool_calls, reasoning) 将 tool_calls 转为可序列化 dict 后追加

适配统一 ToolCall 格式和旧 SDK 对象两种情况。

public accumulate_usage(usage, is_tool_round, elapsed_ms) 累加一轮 LLM 调用的 token 消耗

统计 prompt_tokens、completion_tokens、cached_tokens、task_tokens（工具轮消耗）、总轮数、总耗时。

public save_history() / load_history() 历史消息持久化

save_history: 写入前自动闭合末尾未完成的 tool_calls 链（防 Ctrl+C 中断）。

load_history: 从 JSON 恢复，损坏时自动备份并重置。

private _repair_tool_chain() 修复历史中的 tool_calls 链断裂

三种断裂情况：

  • assistant(tool_calls) 之后缺少 tool 消息（前向断裂）
  • tool 消息之前缺少 assistant(tool_calls)（后向断裂）
  • tool 消息出现在 assistant(tool_calls) 之前（乱序）

检测到断裂 → 切除末尾消息，打印 [历史修复] 日志。

private _trim_if_needed() 超限时裁剪历史，生成摘要，归档旧消息

超过 max_history → 保留后半段 → 提取摘要插入开头 → 旧消息归档（gzip 压缩）。

裁剪点向后扫描，确保不切断 tool_calls/tool_result 配对。

public clear_history() / archive_now() 清除/归档历史

clear_history: 先归档再清空，同时清空工具日志。

archive_now: 手动归档当前全部历史，不清空。

run/tool.py

运行时核心 ./run/tool.py

工具执行 — 注册 / 权限校验 / 限流 / 全局超时 / 日志

public register_tool(schema, handler, meta) 注册工具到全局注册表

TOOL_REGISTRY[name] = (schema, handler, meta)

幂等注册，同名覆盖。

public load_tool_schemas() 返回排序后的 schema 列表（传给 LLM）

→ list[dict]: 所有注册工具的 OpenAI function calling schema

class ToolRunner

工具执行器 — 权限校验 / 限流 / 日志。每个用户一个实例。

max_total — 单轮总调用上限（默认 80）
max_per_tool — 单工具调用上限（默认 80）
tool_timeout — 单次执行超时（默认 120s）
_enabled / _deny — 权限配置（deny 优先）

public execute(response) 执行所有 tool_calls，返回结果和详情

对每个 tool_call：

  • 权限校验 → deny/enabled 检查
  • 限流检查 → 全局/单工具上限
  • 执行 → ThreadPoolExecutor 带超时
  • 日志 → log_tool_call()

→ tuple[list[dict], list[dict]]: (tool result 消息, 调用详情)

public has_tool_calls(response) 检查响应是否包含工具调用

适配 ProviderResponse 和旧 SDK 对象两种格式。

public reset_count() 重置调用计数器（每轮对话开始时调用）

清零 call_count 和 per_tool_count。

run/summarize.py

运行时核心 ./run/summarize.py

对话摘要子代理 — 独立 LLM 调用生成一句话摘要，用于归档索引

public generate_summary(provider, messages) 子代理：用独立 LLM 调用生成 ≤30 字中文摘要

取最近 12 条核心 user/assistant 消息（跳过 tool_calls 和 tool 结果），

使用简短 system prompt 约束输出为中文摘要。

→ str: 摘要文本（≤50 字）

public summarize_and_store(provider, messages, user_dir) 生成摘要并存入索引

即使 LLM 摘要失败，也会写入 msg_count，确保归档同步链路不中断。

摘要存入 history_save_data.json 索引文件。

public sync_to_new_archives(user_dir, before) 将当前摘要同步到新生成的归档文件

/clear 或 /archive 后调用，对比归档目录变化，为新文件写入摘要元数据。

skills/__init__.py

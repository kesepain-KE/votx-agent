votx-agent 运行流程

基于源码分析的完整架构可视化 — Python AI Agent Framework

用户输入

入口层
CLI 模式 — main.py
终端交互，支持流式输出、思考过程展示
Web 模式 — web/server.py
Flask + SSE 事件流，Vue 3 单页前端，多用户 session 隔离
🔒

Session 隔离

web/session.py
按 user_name 分桶隔离，每个用户独立的配置、对话历史、Provider 实例。 Web 面板切换用户时重建 Provider（type/api_style/api_key/base_url 变更触发）。
🧩

System Prompt 组装

run/engine.py — build_system_prompt()
从多个源文件按优先级叠加组装完整 system prompt，由 prompt_cache.py 按 mtime 缓存：
users/<name>/self_soul.md — 用户人设（最外层，优先级最高） P0
config/soul.md — 全局人格基座 P1
AGENTS.md — 自我操作手册（架构、规则、Skill 目录） P2
skills.register_all() — 动态 Skill 摘要（12 工具型 + 15 指令型） P3
self-improving/memory.md — HOT Tier 自改进记忆 P4
self-improving/corrections.md — 纠正记录 P5
memory/*.md — 长期记忆（跨会话持久化，单文件限 2000 字符） P6
memory/ontology/graph.jsonl — 知识图谱摘要（仅注入实体统计） P7
SESSION-STATE.md — 会话级状态（Hot RAM） P8
🔌

Provider 创建

provider/factory.py
根据 config.json 的 provider.type 创建对应实例，所有厂商通过 adapter 转换为统一的 ProviderResponse 格式：
OpenAI → ResponsesProvider
DeepSeek → ResponsesProvider
Azure → ResponsesProvider
Gemini → ResponsesProvider
Anthropic → AnthropicProvider
api_style 控制协议："responses" (OpenAI Responses API) / "chat" (Chat Completions)
🔄

对话轮次 — Tool Calling 循环

engine.py — run_chat_turn()
核心执行循环，CLI 和 Web 共用：
🔁 while tool_round < MAX_TOOL_ROUNDS (默认 80)
Step 1. chat.build_messages() 构建消息列表
Step 2. 调用 Provider（流式 / 非流式）
Step 3. 检查 response.tool_calls
有
tool_calls?
→ Yes 执行工具 → 结果追加到消息 → 回到 Step 1
→ No 这是最终回复 → 结束循环
⚠️ 死循环检测：同一命令连续失败 3 次 → 警告并停止重试
⚠️ 超过 MAX_TOOL_ROUNDS → 强制终止，提示用户重新描述

🗑️ delete_file

📤 http_post
⚡ run_command
🔍 tavily_search

📄 read_docx
🕐 get_time
⏰ sleep
🧠 mem_save/get
🔮 ontology_*
🖼️ generate_image

响应输出

事件流
事件类型
text_chunk — 流式文本
thinking_chunk — 思考过程（DeepSeek/Claude）
tool_call — 工具调用进度
usage — Token 用量统计
error — 错误信息
输出路径
CLI → 终端流式打印
Web → SSE 事件流推送到前端
工具产物 → users/<name>/download/
临时文件 → tmp/
🧠

自改进闭环

skills/self-improving/
分层记忆
🔥 HOT — 用户偏好、模式、规则（注入 system prompt）
🌡️ WARM — 会话级上下文
❄️ COLD — 归档的历史对话
闭环机制
被纠正 → 记录到 corrections.md
自我反思 → 更新 memory.md
知识积累 → ontology/graph.jsonl
下次对话自动注入 → 越用越聪明

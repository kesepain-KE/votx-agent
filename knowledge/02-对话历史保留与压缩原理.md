# 对话历史保留与压缩原理

## 一、历史存储结构

```
users/<name>/history/
├── log/                    ← 完整日志（含工具调用中间步骤）
│   └── *.jsonl             ← 每次对话 append 一行 JSON
├── archive/                ← 定时任务归档
│   └── cron_*.json
├── file/                   ← 用户文件池（上传、外部附件）
└── conversations.jsonl     ← 会话索引（conversation_id, 标题, 时间）
```

## 二、ChatManager 消息模型

```python
# run/chat.py
class ChatManager:
    def __init__(self, user_dir, history_rounds=20):
        self.messages: list[dict] = []        # 完整历史
        self.system_prompt: str = ""          # 当前 system prompt
        self._history_rounds = history_rounds  # 保留轮数
```

消息结构（每条）：
```json
{
  "role": "user" | "assistant" | "tool",
  "content": "消息文本",
  "tool_calls": [{"id": "call_xxx", "type": "function", "function": {"name": "...", "arguments": "..."}}],
  "tool_call_id": "call_xxx",
  "name": "tool_name"
}
```

## 三、保留原理

### 加载时截断

```python
def load_history(self):
    """从 log/*.jsonl 加载最近 N 轮对话"""
    # 读取最近的 log 文件
    # 按 conversation_id 过滤
    # 只保留最近 _history_rounds 轮（默认 20 轮）
    # 一轮 = user 消息 + assistant 回复
```

### 保留规则
- 默认保留最近 **20 轮**对话（`history_rounds` 配置）
- 每轮包含：user 消息 + assistant 回复（含工具调用和结果）
- 超出轮数的历史不加载到内存，但 log 文件中仍完整保留
- system prompt 不计入轮数，始终保留

### build_messages() 输出

```python
def build_messages(self) -> list[dict]:
    """构建发送给 LLM 的完整消息列表"""
    result = []
    result.append({"role": "system", "content": self.system_prompt})
    recent = self.messages[-(self._history_rounds * 2):]
    result.extend(recent)
    return result
```

## 四、压缩原理

### 触发条件
当 `build_messages()` 的 token 数超过 `context_window` 的阈值时（通常 75%~80%），触发自动压缩。

### 压缩策略

```python
# run/history.py - compress_history()
def compress_history(messages: list[dict], provider, max_tokens: int) -> list[dict]:
    """压缩历史消息以适应上下文窗口"""
```

压缩流程：
1. **计算当前 token 数** — 通过 provider 的 tokenizer
2. **保留边界**：
   - system prompt 始终保留（不压缩）
   - 最近 2~4 轮对话保留原样（保证连贯性）
3. **中间消息压缩**：
   - 将中间的历史消息合并为一段摘要
   - 使用 LLM 生成摘要（`provider.respond()`）
   - 摘要格式：`[历史摘要] 之前对话中讨论了...用户要求...AI完成了...`
4. **替换**：用摘要替换原始中间消息

### 压缩后消息结构

```
[system]  完整 system prompt（不压缩）
[摘要]    [历史摘要] 之前对话讨论了 X、Y、Z...
[user]    第 N-1 轮用户消息（保留原样）
[asst]    第 N-1 轮 AI 回复（保留原样）
[user]    第 N 轮用户消息（当前轮）
```

### 压缩时机
- 在 `run_chat_turn()` 开始前检查
- 如果 token 数超限，先压缩再发送请求
- 压缩发生在内存中，不影响磁盘上的 log 文件

## 五、保存原理

### 保存时机
- 每次 `run_chat_turn()` 结束后调用 `chat.save_history()`
- 每条消息按 JSONL 格式 append 到 log 文件

### 保存内容
- 完整消息（含工具调用中间步骤）
- 不截断、不压缩
- 包含时间戳、conversation_id

## 六、外部消息历史

外部消息（QQ/Telegram）通过 `AgentService.chat()` 调用，历史与 Web/CLI 共享同一个 `ChatManager`：
- 同一用户的所有渠道共享历史
- source 前缀作为 user 消息的一部分保存
- 外部消息的附件路径也保存在消息中

## 七、关键配置

| 配置项 | 位置 | 默认值 | 说明 |
|--------|------|--------|------|
| `history.rounds` | `users/<name>/config.json` | 20 | 保留轮数 |
| `history.compress_threshold` | `config.json` | 0.75 | 压缩触发阈值 |
| `history.log_format` | `config.json` | jsonl | 日志格式 |

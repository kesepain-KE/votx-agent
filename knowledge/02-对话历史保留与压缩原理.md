# 对话历史保留与压缩原理

更新时间：2026-07-12

## 存储

用户历史目录：

```text
users/<name>/history/
├── chat/       当前聊天数据
├── log/        对话、工具和附件日志
├── archive/    归档历史
└── file/       用户上传与外部附件
```

具体文件名来自用户 `config.json` 的 `history.data` 和 `history.log`。`ChatManager.messages` 使用 OpenAI 风格的 user/assistant/tool 消息，并保留 tool_calls、tool_call_id 和可选 reasoning 内容。

## 三条压缩路径

### 条数自动压缩

`add_user_message()` 和 `add_tool_results()` 调用 `_trim_if_needed(force=False)`。当消息数达到 `config/config_core.json` 的 `history.history_max` 时：

1. 选择较早的一段历史。
2. 调整切点，避免切断 assistant tool_calls 与后续 tool 消息。
3. 生成 `[历史压缩]` 摘要放在保留历史前。
4. 修复工具链结构。
5. 触发被动 auto_improve。
6. 设置前端压缩状态标记。

当前全局默认 `history_max` 为 100，实际值以配置文件为准。

### 手动 `/compress`

Web/CLI 命令调用 `chat.compress_history()`，复用 `_trim_if_needed(force=True)`：

- 不等待 history_max 达标。
- 至少需要 4 条消息。
- 压缩较早部分并至少保留 2 条近期消息。
- 成功后立即 `save_history()`。
- 返回压缩前后消息数；失败返回明确错误。

### Token 预算压缩

`build_messages()` 调用 `_ensure_token_budget()`，预算来自：

```text
context_window.max_tokens × context_window.safe_ratio
```

当前全局默认值为 1,000,000 × 0.85。Token 数由本地字符启发式估算，不依赖具体厂商 tokenizer，因此 `/compress` 提供人工提前压缩入口。

超过预算时会保留 System Prompt 和近期历史，按安全切点删除较早内容并加入摘要。其目标是发给 Provider 的消息不超预算；历史持久化由对话结束保存流程完成。

## Web 状态

自动压缩通过 `_compress_occurred` 触发 SSE `ui_status`。手动 `/compress` 由前端命令状态显示“正在压缩上下文...”，执行期间阻止重复提交，完成后清除状态。

## 保存

- `save_history()` 将当前 `chat.messages` 原子写入聊天数据文件。
- `save_log()` 写入用于查看的对话日志。
- Web 对话结束时会在 finally 流程保存，避免异常导致历史丢失。
- 手动 `/compress` 在命令处理器内立即保存。

外部 QQ/Telegram 消息按同一内部用户共享 ChatManager 历史，因此使用相同压缩机制。

## 关键配置

| 配置 | 来源 |
|---|---|
| `history.history_max` | `config/config_core.json` |
| `context_window.max_tokens` | `config/config_core.json` |
| `context_window.safe_ratio` | `config/config_core.json` |
| 历史文件名/开关 | `users/<name>/config.json` 的 `history` |
